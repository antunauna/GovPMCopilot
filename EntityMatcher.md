## Skill 定位

EntityMatcher 是 GovPMCopilot 系统中的**主体资格匹配核心模块**。

其核心目标是：给定一个或多个政府项目/政策，自动评估集团旗下各法人主体的申报资格，输出匹配评分、缺口分析和优先推荐主体，帮助项目经理快速锁定最优申报主体，避免资格误判和资源浪费。

---

## 触发场景

当用户输入以下任意内容时，激活本 Skill：

- "这个项目哪个主体可以报？"
- "XX 公司能不能申报这个政策？"
- "帮我看看哪个子公司最适合报这个项目"
- "对比一下我们几个主体，谁最符合申报条件"
- "XXX 政策的申报门槛我们能达标吗？"
- 输入政策文本 + 企业信息，要求进行资格评估

---

## 输入规范

### 必填输入

#### 1. 政策/项目申报条件（至少提供其中一种）

```
- 政策原文或摘要
- 申报指南链接
- 已结构化的政策对象（JSON）
```

#### 2. 企业主体信息（至少提供其中一种）

```
- 企业基础资料描述（自然语言）
- 结构化主体画像（JSON）
- 多主体列表（批量匹配）
```

### 可选输入

- 匹配优先级偏好（如：优先考虑资金体量 / 优先考虑胜率）
- 排除主体列表
- 历史申报记录（用于避免重复申报检测）

## 核心处理逻辑

### 第一阶段：政策条件结构化解析

从政策原文中自动提取以下申报门槛字段：

```
硬门槛（规则引擎强校验）：
  - 企业类型限制
  - 注册地要求
  - 成立年限
  - 营收规模
  - 研发费用占比
  - 员工人数
  - 知识产权数量要求
  - 资质标签（高企/专精特新/规上/创新型中小企业等）
  - 财务健康状态（无重大违规/纳税信用）

软指标（模型辅助评估）：
  - 技术方向契合度
  - 项目实施能力
  - 历史申报成功率
  - 材料完备程度预估
  - 地方关系与配套资源
```

### 第二阶段：主体画像标准化

对每个候选主体，结构化提取以下画像字段：

```json
{
  "entity_id": "ENT_001",
  "name": "XX科技有限公司",
  "region": "广东省深圳市",
  "industry": "电子信息/智能制造",
  "established_years": 8,
  "revenue_last_year": 23000000,
  "rd_ratio": 0.062,
  "employee_count": 320,
  "qualifications": ["高新技术企业", "专精特新"],
  "patents": {"invention": 12, "utility": 28, "design": 5},
  "software_copyright": 15,
  "tax_credit": "A级",
  "history_projects": ["2023年省级智能制造项目", "2022年市级研发专项"],
  "financials_updated": "2025-12-31",
  "risk_flags": []
}
```

### 第三阶段：规则引擎硬校验

对每个硬门槛条件执行严格的规则判断：

```
IF 主体.注册地 NOT IN 政策.适用地区 → ❌ 直接不推荐
IF 主体.成立年限 < 政策.最低成立年限 → ❌ 直接不推荐
IF 主体.营收 < 政策.最低营收要求 → ❌ 直接不推荐
IF 主体.研发占比 < 政策.最低研发占比 → ⚠️ 标注缺口
IF 主体.资质 NOT IN 政策.要求资质 → ❌ 直接不推荐
```

> ⚠️ 重要约束：硬门槛校验结果不得由大模型覆盖或解释性通过，必须严格执行规则。

### 第四阶段：软指标模型评分

对通过硬门槛的主体，使用大模型进行软指标综合评分：

```
评分维度（满分100）：
  硬门槛达标度    40分（规则引擎）
  技术契合度      20分（模型评估）
  项目实施能力    15分（模型评估）
  材料完备预估    15分（模型评估）
  历史申报成功率  10分（历史数据）
```

### 第五阶段：缺口分析与修复建议

对评分在 50-79 分的"条件推荐"主体，输出具体缺口和可行的补足建议：

```
示例：
缺口：研发费用占比 2.1%，距要求 3% 差 0.9%
建议：
  1. 若当年可调整费用归集口径，补足约 XXX 万元研发投入
  2. 若无法补足，建议改由其他主体申报
  3. 或等下一财年指标达标后再行申报
```

---

## 使用约束与合规要求

```
1. 硬门槛结论必须来自政策原文或结构化数据，不得推测
2. 所有评分结论必须附带数据来源和依据原文
3. 当主体数据缺失关键字段时，必须标注"待补充/需进一步验证"
4. 不得为不满足硬门槛的主体出具"可以尝试申报"的结论
5. 涉及财务数据、知识产权数量等关键指标，必须以企业提供数据为准，不得推测填充
6. 重复申报检测为必选步骤，不可跳过
7. 最终申报决策须由人工确认，本 Skill 输出仅供参考
```
