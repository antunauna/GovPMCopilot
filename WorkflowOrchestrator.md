## Skill 定位

WorkflowOrchestrator 是 GovPMCopilot 系统的**中枢调度与流程编排模块**。

其核心目标是：将 PolicyRadar、EntityMatcher、PolicyInterpreter、ProposalComposer、ReviewAuditor 五个子 Skill 串联为一条完整的自动化工作流，负责任务调度、节点流转、状态管理、超时告警、人工确认节点介入、版本管理和全链路审计追踪，确保从"新政策发现"到"材料最终提交"的全链路高效、可控运转。

---

## 触发场景

当用户输入以下任意内容时，激活本 Skill：

- "启动一个新的申报项目"
- "帮我跑完整个申报流程"
- "从政策发现到材料生成全部帮我自动化"
- "查看当前申报项目的进度"
- "这个项目卡在哪一步了？"
- "帮我催一下XX节点"
- "给我看所有进行中的申报项目"
- "生成本月申报项目进度报告"
- 系统检测到新政策入库，自动触发流程初始化

## 输入规范

### 流程启动输入

```
- 触发方式：新政策入库自动触发 / 用户手动发起
- 政策信息：PolicyRadar 推送的政策对象 / 用户手动输入
- 企业范围：指定参与匹配的主体列表 / 默认全量主体
- 流程模式：全自动模式 / 半自动模式（关键节点人工确认）
- 优先级：普通 / 重要 / 紧急（影响资源调度顺序）
```

### 流程查询输入

```
- 项目 ID / 政策名称 / 申报主体名称
- 查询维度：进度 / 问题 / 待办 / 历史记录
- 时间范围：当前 / 本周 / 本月 / 自定义
```

## 核心工作流定义

### 主流程：政策发现 → 材料提交

```
┌─────────────────────────────────────────────────────────┐
│                  GovPMCopilot 主工作流                    │
└─────────────────────────────────────────────────────────┘

[节点 1] 政策发现与入库
PolicyRadar
  → 抓取新政策
  → 结构化解析
  → 入库政策知识库
  → 触发主流程
  ↓
  ✅ 自动执行
  ⏱️ 超时：24小时未完成 → 告警

[节点 2] 主体资格匹配
EntityMatcher
  → 获取全量企业主体画像
  → 执行硬门槛校验
  → 生成匹配评分与排名
  → 输出推荐主体列表
  ↓
  🔵 人工确认节点
  → 项目经理确认/调整推荐主体
  → 超时 48 小时未确认 → 自动提醒 → 继续等待
  → 确认后进入下一节点

[节点 3] 政策深度解读
PolicyInterpreter
  → 基于确认主体生成定向解读
  → 输出六维解读报告
  → 生成推荐申报项目池
  ↓
  🔵 人工确认节点
  → 项目经理确认申报方向与项目名称
  → 超时 48 小时未确认 → 自动提醒
  → 确认后进入下一节点

[节点 4] 材料收集
WorkflowOrchestrator（本节点）
  → 生成材料收集清单
  → 向相关责任人发送收集任务
  → 追踪材料上传进度
  ↓
  🔵 人工操作节点
  → 责任人上传所需材料
  → 超时 72 小时未完成 → 升级告警
  → 材料齐全后自动进入下一节点

[节点 5] 申报材料撰写
ProposalComposer
  → 融合已收集材料 + 解读报告
  → 生成申报材料初稿
  → 输出待填充项清单
  ↓
  🔵 人工确认节点
  → 项目经理补全占位符
  → 确认材料内容真实性
  → 超时 72 小时未确认 → 自动提醒

[节点 6] 预审复核
ReviewAuditor
  → 执行六维预审
  → 生成问题清单与修改建议
  → 输出综合评分
  ↓
  ⚙️ 自动判断节点
  IF 评分 ≥ 90 且 无 P0 问题
    → 自动进入提交准备节点
  IF 评分 75-89 或 P0 问题 ≤ 2
    → 返回节点 5 定向修改
  IF 评分 < 75 或 P0 问题 ≥ 3
    → 🔵 人工介入：项目经理决定修改方案
  → 修改完成后重新触发节点 6（最多循环 3 次）

[节点 7] 提交准备与终审
WorkflowOrchestrator（本节点）
  → 汇总全套材料
  → 执行最终一致性检查
  → 生成提交版本包
  → 生成提交检查清单
  ↓
  🔴 人工终审节点（必须）
  → 项目负责人 + 法务/财务 确认签字
  → 确认后标记"可提交"状态
  → 超时 24 小时未确认 → 紧急告警

[节点 8] 提交归档
WorkflowOrchestrator（本节点）
  → 记录提交时间、版本、负责人
  → 归档全套材料至项目台账
  → 更新项目状态为"已提交"
  → 设置后续跟踪提醒（评审结果、补充材料等）
  ↓
  ✅ 流程完成
```

---

## 节点状态定义

| 状态码 | 状态名称 | 说明 |
|--------|----------|------|
| `PENDING` | 待执行 | 节点尚未开始 |
| `RUNNING` | 执行中 | 节点正在自动处理 |
| `WAITING_HUMAN` | 等待人工 | 节点需要人工确认或操作 |
| `TIMEOUT_WARNING` | 超时预警 | 节点执行时间超过阈值 |
| `TIMEOUT_ESCALATED` | 超时升级 | 超时预警后仍未处理，升级告警 |
| `COMPLETED` | 已完成 | 节点成功完成 |
| `FAILED` | 失败 | 节点执行失败，需人工介入 |
| `SKIPPED` | 已跳过 | 节点被人工跳过（需记录原因） |
| `ROLLED_BACK` | 已回退 | 节点被回退至上一步 |

```

✅ 法定代表人身份证复印件
✅ 上年度经审计财务报表
✅ 纳税信用等级证明
✅ 无违规承诺书（法人签字）
```

```
科技类项目追加：
  ✅ 高新技术企业证书（如有）

  ✅ 专精特新证书（如有）
  ✅ 规上企业认定证明（如有）
  ✅ 主要设备清单及发票

  ✅ 银行开户许可证
  ✅ 近3年完税证明
  ✅ 社会保险缴纳证明
```

## 版本管理机制

```
每次材料修改自动生成新版本：

v1.0 — ProposalComposer 初稿生成
v1.1 — 项目经理补全占位符
v1.2 — ReviewAuditor 预审后第一轮修改
v1.3 — ReviewAuditor 预审后第二轮修改
v2.0 — 终审确认版本（提交版）

版本管理规则：
  - 每个版本自动保存，不可覆盖删除
  - 支持任意版本对比查看
  - 提交版本必须标注 v2.0 或以上
  - 版本回退需记录原因和操作人
```

## 重复申报检测机制

```
在节点2（主体匹配）执行时，自动执行重复申报检测：

检测维度：
  1. 同一主体 × 同类政策（近3年内）
  2. 同一建设内容 × 不同政策（近3年内）
  3. 同一设备清单 × 不同项目
  4. 同一知识产权 × 多个项目同时使用

检测结果处理：
  ✅ 无重复风险 → 正常推进
  ⚠️ 存在潜在重复 → 生成风险提示，需人工确认
  ❌ 确认重复申报 → 阻断流程，禁止继续
```

## 申报截止日期预警机制

```
WorkflowOrchestrator 持续监控所有进行中项目的申报截止日期：

距截止 30 天：生成项目进度检查提醒
距截止 14 天：升级为重要提醒，推送给项目负责人
距截止 7 天：🔴 紧急告警，推送给部门负责人
距截止 3 天：🚨 最高级告警，推送给所有相关人
截止日当天：锁定提交窗口，记录最终状态
```

## 全链路审计追踪

WorkflowOrchestrator 对所有操作自动记录审计日志：

```
审计日志字段：
  - 操作时间
  - 操作人
  - 操作类型（确认/修改/回退/跳过/提交）
  - 操作节点
  - 操作前状态
  - 操作后状态
  - 操作备注

审计日志用途：
  - 申报合规性留存
  - 问题追溯与复盘
  - 团队绩效分析
  - 监管检查应对
```

## 与其他 Skill 的协作关系

```
┌─────────────────────────────────────────────────────┐
│              WorkflowOrchestrator                    │
│                  （中枢调度）                         │
└──────┬──────┬──────┬──────┬───────────────────────┘
       ↓      ↓      ↓      ↓
  PolicyRadar  EntityMatcher  PolicyInterpreter
  （节点1）     （节点2）       （节点3）
                                    ↓
                             ProposalComposer
                               （节点5）
                                    ↓
                             ReviewAuditor
                               （节点6）
                                    ↓
                          WorkflowOrchestrator
                          （节点4/7/8 自执行）
```

### 各 Skill 调用关系

| 节点 | 调用 Skill | 调用方式 | 输出去向 |
|------|-----------|----------|----------|
| 节点1 | PolicyRadar | 定时触发/手动触发 | 节点2输入 |
| 节点2 | EntityMatcher | 自动调用 | 人工确认后入节点3 |
| 节点3 | PolicyInterpreter | 自动调用 | 人工确认后入节点4 |
| 节点4 | 自执行 | 清单生成+进度追踪 | 材料齐全后入节点5 |
| 节点5 | ProposalComposer | 自动调用 | 人工确认后入节点6 |
| 节点6 | ReviewAuditor | 自动调用 | 评分判断后路由 |
| 节点7 | 自执行 | 汇总+一致性检查 | 人工终审后入节点8 |
| 节点8 | 自执行 | 归档+台账更新 | 流程结束 |

---

## 使用约束与合规要求

```
1. 节点7（终审确认）为强制人工节点，不可跳过或自动通过
2. 一票否决项（合规红线）触发后，流程自动阻断，不可强制推进
3. 重复申报检测确认为❌时，流程阻断，需法务确认后方可解锁
4. 所有节点跳过操作必须填写原因，并记录操作人信息
5. 版本回退操作需双人确认（操作人 + 项目负责人）
6. 审计日志不可删除，保存期不少于5年
7. 超时升级告警后，系统自动抄送部门负责人，不可关闭告警
```
